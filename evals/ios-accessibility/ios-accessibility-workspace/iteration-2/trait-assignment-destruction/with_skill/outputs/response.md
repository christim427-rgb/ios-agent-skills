# UIKit Trait Assignment Destruction: Why `= .selected` Breaks VoiceOver

## Severity: 🔴 CRITICAL

This is **AI failure pattern F10** — one of the most common accessibility bugs in UIKit code generated by AI assistants. It silently strips away the element's role, causing VoiceOver to lose important context.

---

## What Happens With `= .selected`

`accessibilityTraits` is a bitmask (`UIAccessibilityTraits`). When you use the assignment operator `=`, you **replace the entire bitmask** with just the one trait you specified.

```swift
// Your toggle button starts with these traits (set by UIKit automatically):
// myToggle.accessibilityTraits == [.button]
// VoiceOver announces: "Subscribe, button"

// ❌ WRONG — assignment DESTROYS existing traits
myToggle.accessibilityTraits = .selected
// myToggle.accessibilityTraits is now ONLY [.selected]
// .button has been erased

// VoiceOver now announces: "Subscribe, selected" — with NO "button" announcement
// The user has lost all knowledge that this is an interactive control
```

The `.button` trait is what tells VoiceOver "this element can be activated with a double-tap." Without it, VoiceOver users may not realize the element is interactive at all. They hear "selected" but get no indication of what type of control they're on.

---

## The Fix: Always Use `.insert()` and `.remove()`

Use the bitmask mutation methods to add or remove individual traits **without disturbing the others**:

```swift
// ✅ CORRECT — insert adds .selected while keeping .button intact
myToggle.accessibilityTraits.insert(.selected)
// myToggle.accessibilityTraits == [.button, .selected]
// VoiceOver announces: "Subscribe, selected, button"

// ✅ CORRECT — remove .selected when toggling off
myToggle.accessibilityTraits.remove(.selected)
// myToggle.accessibilityTraits == [.button]
// VoiceOver announces: "Subscribe, button"
```

## Complete Toggle Pattern

Here is the full pattern for a UIKit custom toggle button that correctly manages traits:

```swift
class FavoriteToggleButton: UIButton {

    var isFavorited: Bool = false {
        didSet {
            updateTraits()
            updateAppearance()
        }
    }

    override init(frame: CGRect) {
        super.init(frame: frame)
        // Set initial traits — UIButton already has .button, add .selected if needed
        updateTraits()
    }

    required init?(coder: NSCoder) {
        super.init(coder: coder)
        updateTraits()
    }

    private func updateTraits() {
        if isFavorited {
            // ✅ Insert .selected — keeps .button intact
            accessibilityTraits.insert(.selected)
        } else {
            // ✅ Remove .selected — keeps .button intact
            accessibilityTraits.remove(.selected)
        }
    }

    private func updateAppearance() {
        setImage(
            UIImage(systemName: isFavorited ? "heart.fill" : "heart"),
            for: .normal
        )
        accessibilityLabel = isFavorited ? "Remove from favorites" : "Add to favorites"
    }

    @objc func handleTap() {
        isFavorited.toggle()
    }
}
```

## When Is Direct Assignment (`=`) Acceptable?

Only when setting traits on a **brand new** element that has no prior traits — typically in `init` or `awakeFromNib` — and you're setting the complete set from scratch:

```swift
// ✅ OK in init — element has no prior traits, set the full initial set
let element = UIAccessibilityElement(accessibilityContainer: self)
element.accessibilityTraits = [.button, .selected]
```

For any **state change** on an existing view — always use `.insert()` / `.remove()`.

---

## Summary

| Operation | Code | Effect |
|---|---|---|
| Assignment | `myToggle.accessibilityTraits = .selected` | Replaces ALL traits — destroys `.button` |
| Insert | `myToggle.accessibilityTraits.insert(.selected)` | Adds to existing traits — safe |
| Remove | `myToggle.accessibilityTraits.remove(.selected)` | Removes one trait — safe |

The rule is simple: **state changes use `.insert()` / `.remove()`, never `=`**.
